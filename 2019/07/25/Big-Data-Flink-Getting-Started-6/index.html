Flink 进阶篇

1. Flink Runtime核心机制剖析

1. 整体架构

  • 物理资源层
  • Runtime统一执行引擎
  • API层
  • High-level API层

运行模式

  • Per-Job
    • 独享Dispatcher与Resource Manager
    • 按需要申请资源(即TaskExecutor)
    • 适合执行时间较长的大作业
  • Session
    • 共享Dispatcher和Resource Manager
    • 共享资源(即TaskExecutor)
    • 适合规模小，执行时间短的作业

2. 资源管理与作业调度

  • 调度：任务与资源匹配
  • 资源：Slot
    • 大小(未启用)
    • Location
    • 每个TaskManager包含一个或多个Slot
  • 任务
    • 大小(未启用)
    • CoLocation Constraint

Slot管理

  • ResourceManager
    • Slot Manager
    • 管理Slot状态
    • 分配Slot资源
  • TaskExecutor
    • 实际持有Slot资源
  • JobMaster
    • Slot资源的申请者

Slot划分

  • 社区版未启用Slot大小
    • TM启动固定个数的Slot
    • 默认每个Task占用一个Slot
      • 可以通过Slot Sharing在单个Slot中启动多个Task
  • Blink开源版
    • 增加Slot资源严格匹配
    • ResourceProfile：(CPU, Heap, …)
    • Per-Job模式：根据Task资源申请合适的Slot资源
    • Session模式：从预先申请好的资源中扣除

Slot Sharing

  • 单个Slot中部署多个Task
    • 同一Vertex的多个Task不能共享Slot

作业DAG图结构

  • JobGraph
    • 客户端提交的作业DAG图结构
    • 逻辑结构，不考虑并发
  • ExecutionGraph
    • JobManager实际维护的数据结构
    • 物理结构，考虑并发

调度

  • Eager调度
    • 适用于流作业
    • 一次性调度所有的Task
  • LAZY_FROM_SOURCE
    • 适用于批作业
    • 上游作业执行完成后，调度下游的作业

3. 错误恢复

  • Task Failover
    • 单个Task执行失败或TM出错退出等
    • 可以有多种不同的恢复策略
  • Master Failover
    • AM执行失败

Task Failover:Restart-all

  • 重启所有Task
    • 从上次的Checkpoint开始重新执行

Task Failover:Restart-individual

  • 只重启出错的Task
    • 只能用于Task间无连接的情况
    • 应用极为有限

Task Failover:Restart-Region

  • 重启Pipeline Region
    • Blocking数据落盘，可以直接读取
    • 逻辑上仅需重启通过Pipeline边关联的Task
  • 两种类型错误
    • 作业自身执行失败
    • 作业读取上游数据失败

Master Failover

  • 多个Master通过ZK进行选主
  • 目前Master Failover要求全图重启

4. 未来展望

  • 完善的资源管理
  • 同一Stream和Batch
  • 更灵活的调度策略
  • Master Failover优化


2. Flink Time 深度解析

1. Flink时间语义

  • 时间不是绝对的，不同”世界“中的时间语义各不相同
  • Processing Time
    • 真实世界的时间
    • 处理数据节点的本地时间
    • 处理简单
    • 结果不确定(无法重现)
  • Event Time(Row Time)
    • 数据世界的时间
    • 记录携带的Timestamp
    • 处理复杂
    • 结果确定(可重现)

时间的特性

  • Processing Time：递增
  • Event Time：一定程度的乱序

2. Timestamp和Watermark

Timestamp分配和Watermark生成

  • 在SourceFunction中产生
    • collectWithTimestamp(T element, long timestamp)
    • endWatermark(Watermark mark)
  • 在流程中指定
    • DataStream.assignTimestampAndWatermarks(…)
    • 定期生成
      • 现实时间驱动
      • 每隔一段时间调用生成方法
      • 实现AssignerWithPeriodicWatermarks
    • 根据特殊记录生成
      • 数据驱动
      • 每一次分配Timestamp都会调用生成方法
      • 实现AssignerWithPunctuatedWatermarks

Watermark传播

  • Watermark以广播的形式在算子之间进行传播
  • Long.MAX_VALUE表示不会再有数据
  • 单输入取其大，多输入取其小
  • 局限：没有区分逻辑上的单流和多流，强制同步时钟

ProcessFunction

  • 获取记录的Timestamp或当前ProcTime
  • 获取算子时间(Watermark)
  • 注册Timer并提供回调逻辑
    • registerEventTimeTimer()
    • registerProcessionTimeTimer()
    • onTime()

Watermark处理

  • 更新算子时间
  • 便利计时器队列触发回调
  • 将Watermark发送至下游

3. Table中的时间

Table中指定时间列

  • Processing Time
    • 从DataStream转化
      • tEnv.fromDataStream(stream, “f1, f2, f3.proctime”)
    • 通过TableSource生成
      • TableSource实现DefinedProctimeAttributes接口
  • Event Time
    • 从DataStream转化
      • 原始DataStream必须有Timestamp及Watermark
      • tEnv.fromDataStream(stream, “f1, f2, f3.proctime”)
      • tEnv.fromDataStream(stream, “f1.proctime, f2, f3”)
    • 通过TableSource生成
      • 数据中存在类型为long或timestamp的时间字段
      • TableSource实现DefinedRowtimeAttributes接口

时间列和Table操作

  • Over窗口聚合(Over Window Aggregation)
  • Group By窗口聚合(Group By Window Aggregation)
  • 时间窗口连接(Time-Windowed Join)
  • 排序(Order By)

4. 深入思考

数据 or 元数据？

  • DataStream API中，时间是元数据
  • SQL/Table API中，时间是数据
  • 在Upsert Stream中不允许出现时间字段

干涉Watermark传播

  • 在发送Watermark之前减去一个固定值

统一EventTime和ProcTime处理

时间和Watermark本质

  • 流处理中时间本质上就是一个普通的递增字段，不一定真的表示时间
  • Watermark只是应对乱序的办法之一，大多是启发式的，在延迟和完整性之间抉择


3. Flink Checkpoint-轻量级分布式快照

1. 总览：Checkpoint与state

  • Checkpoint，在Flink中是一个执行操作，最后产生的结果作为分布式快照提供容错机制
  • State是构成Checkpoint的数据构成

2. 什么是state

  • State：流式计算中持久化了的状态

Keyed State

  • 只能应用于KeyedStream的函数与操作中，例如Keyed UDF, window state
  • Keyed State是已经分区/划分好的，每一个key只能属于某一个keyed State

Operator State

  • 又称为non-keyed state，每一个operator state都仅与一个operator的实例绑定
  • 常见的operator state是source state，例如记录当前source的offset

3. Checkpoint的执行机制

State的存储

Checkpoint执行流程

  • checkpoint coordinator向所有source节点trigger checkpoint
  • 当task收到所有barrier之后，会执行快照，向自己的输出基础传递barrier，并将自己的状态(异步)写入到持久化存储中
  • 当task完成备份后，会将备份数据的地址(state handle)通知给checkpoint coordinator
  • 为了实现EXACTLY ONCE语义，Flink通过一个input buffer将在对齐阶段收到的数据缓存起来，等对齐完成之后再进行处理
  • 对于AT LEAST ONCE语义，无需缓存收集到的数据，会对后续直接处理，所以导致restore时，数据可能会被多次处理
  • Flink的checkpoint机制只能保证Flink的计算过程可以做到EXACTLY ONCE，end-to-end的EXACTLY ONCE需要source和sink的支持

基于RocksDB的增量checkpoint

  • 本地snapshot目录创建当前DB内容的备份
  • 与上一次成功的checkpoint本地sst文件列表比对，将不在其中的文件上传到DFS中
  • 所有文件都会重命名防止冲突
  • 包含了所有新旧文件的handle返回给checkpoint coordinator

从已停止的作业进行状态恢复

  • Savepoint
    • 用户通过命令触发，由用户管理其创建与删除
    • 标准化格式存储，允许作业升级或者配置变更
    • 用户在恢复时需要提供用于恢复作业状态的savepoint路径
  • Externalized Checkpoint
    • Checkpoint完成时，在用户给定的外部持久化存储保存
    • 当作业FAILED(或者CANCELED)时，外部存储的checkpoint会保留下来
    • 用户在恢复时需要提供用于恢复的作业状态的checkpoint路径

类型声明

    • 对于非泛型类，直接传入class对象即可
    • 对于泛型类，需要通过TypeHint来保存泛型类型信息

注册子类型

Env.registerType(typeClass);
  • 获取到的类型信息属于PojoTypeInfo及其子类，那么将会注册到一起，否则统一交给Kryo去处理，Flink并不过问(这种情况下性能会变差)

Kryo序列化

  • 对于Flink无法序列化的类型，默认会交给Kryo处理，如果Kryo仍然无法处理，有两种解决方案：
    • 强制使用Avro来代替Kryo
    • 为Kryo增加自定义的Serializer以增强Kryo的功能

3. Flink通信层的序列化

  • Flink的Task之间如果需要跨网络传输数据记录，那么就需要将数据序列化之后写入NetworkBufferPool，然后下层的Task读出之后再进行反序列化操作，最后进行逻辑处理
  • 为了使得记录以及事件能被写入Buffer随后在消费时再从Buffer中读出，Flink提供了数据记录序列化器(RecordSerializer)与反序列化器(RecordDeserializer)以及时间序列化器(EventSerializer)
  • 我们的Function发送的数据被封装成SerializationDelegate，它将任意元素公开为IOReadableWritable以进行序列化，通过setinstance()来传入要序列化的数据

6. Flink 作业执行解析

1. Flink 4层转化流程

  • 第一层：Program -> StreamGraph
  • 第二层：StreamGraph -> JobGraph
  • 第三层：JobGraph -> ExecutionGraph
  • 第四层：ExecutionGraph -> 物理执行计划

如何转换成StreamGraph

  • 从StreamExecutionEnvironment.execute开始执行程序，将transform添加到StreamExecutionEnvironment的transformations
  • 调用StreamGraphGenerator的generateInternal方法，遍历transformations构建StreamNode及StreamEdge
  • 通过StreamEdge连接StreamNode

StreamNode及StreamEdge

public class WindowWordCount {

    public static void main(String[] args) throws Exception {

        final ParameterTool params = ParameterTool.fromArgs(args);

        // set up the execution environment
        final StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // get input data
        DataStream<String> text = env.readTextFile(params.get("input")).setParallelism(2);

        // make parameters available in the web interface
        env.getConfig().setGlobalJobParameters(params);

        final int windowSize = params.getInt("window", 10);
        final int slideSize = params.getInt("slide", 5);

        DataStream<Tuple2<String, Integer>> counts =
                // split up the lines in pairs (2-tuples) containing: (word,1)
                text.flatMap(new Tokenizer()).setParallelism(4).slotSharingGroup("flatMap_sg")
                        // create windows of windowSize records slided every slideSize records
                        .keyBy(0)
                        .countWindow(windowSize, slideSize)
                        // group by the tuple field "0" and sum up tuple field "1"
                        .sum(1).setParallelism(3).slotSharingGroup("sum_sg");

        // emit result
        counts.print().setParallelism(3);

        // execute program
        env.execute("WindowWordCount");
    }

    public static final class Tokenizer implements FlatMapFunction<String, Tuple2<String, Integer>> {

        @Override
        public void flatMap(String value, Collector<Tuple2<String, Integer>> out) {
            // normalize and split the line
            String[] tokens = value.toLowerCase().split("\\W+");

            // emit the pairs
            for (String token : tokens) {
                if (token.length() > 0) {
                    out.collect(new Tuple2<>(token, 1));
                }
            }
        }
    }
}

7. Flink网络流控及反压剖析

1. 网络流控的概念与背景

为什么需要网络流控

  • 5秒钟后，将面临如下两种情况之一：
    • bounded receive buffer：consumer丢弃新到达的数据
    • unbounded receive buffer：buffer持续扩张，耗尽consumer内存

网络流控的实现：静态限速

  • 静态限流有两点限制：
    • 通常无法实现预估consumer端能承受的最大速率
    • consumer承受能力通常会动态的波动

网络流控的实现：动态反馈/自动反压

  • 动态反馈分两种，广义上的反压机制都涵盖：
    • 负反馈：接收速率小于发送速率时发生
    • 正反馈：发送速率小于接收速率时发生
